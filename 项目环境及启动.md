# 项目环境及启动

本文档说明当前 `xhsocr` 项目在本地运行所需的环境、初始化步骤、数据库迁移方式，以及 API / worker / frontend 的启动命令。

本文档基于当前仓库实际代码整理，适用于当前开发环境与本机运行方式。

## 1. 项目结构

项目根目录：

```text
xhsocr/
├── backend/    # FastAPI + SQLAlchemy + Alembic + RQ Worker
├── frontend/   # Next.js 前端
├── storage/    # 本地存储目录（书稿、任务图片等）
├── docs/       # 文档
├── scripts/    # 脚本
└── README.md
```

## 2. 需要安装的环境

项目根目录提供了模板：

```bash

```

说明：

- 后端配置类会读取 `backend/.env` 吗：当前代码实际读取的是**运行目录中的 `.env`**

  - 直接在终端 `export`
  - 或者在 `backend/.env` 中维护一份

为了减少歧义，推荐你在启动终端里直接 `export` 关键变量。

## 5. 当前项目关键环境变量

下面是当前项目实际用到的重点变量。

### 5.1 后端基础配置

```bash
export APP_ENV=local
export APP_DEBUG=true
export API_HOST=0.0.0.0
export API_PORT=8000
```

### 5.2 数据库与 Redis

```bash
export DATABASE_URL='postgresql+psycopg://xhsocr:change_me@127.0.0.1:5432/xhsocr'
export REDIS_URL='redis://127.0.0.1:6379/0'
```

### 5.3 OCR 相关

当前项目默认推荐：

```bash
export OCR_PROVIDER=rapidocr_onnxruntime
export OCR_TIMEOUT_SECONDS=90
export OCR_ISOLATE_SUBPROCESS=true
export OCR_ALLOW_UNSTABLE_RUNTIME=false
export OCR_AUTO_DOWNGRADE_MACOS=true
```

说明：

- `OCR_PROVIDER=rapidocr_onnxruntime`：当前本机推荐方案，稳定性优先
- `OCR_PROVIDER=paddleocr`：可选，但在 macOS 上稳定性较差，历史上容易崩
- `OCR_PROVIDER=mock`：仅联调用

### 5.4 LLM 相关

当前项目支持通过 OpenRouter 调用模型。

```bash
export LLM_PROVIDER=openrouter
export OPENROUTER_BASE_URL='https://openrouter.ai/api/v1'
export OPENROUTER_API_KEY='你的可用 key'
export LLM_TIMEOUT_SECONDS=300
export LLM_RETRY_COUNT=1
export LLM_RETRY_BACKOFF_SECONDS=1
```

说明：

- 现在**实际生效模型**优先取平台“Prompt 配置中心 / 全局模型配置”
- 启动时 `OPENROUTER_MODEL` 更接近默认值/兜底值
- 如果平台已经配置了当前启用模型，那么新建任务和新重试任务会使用平台里的模型

如果你仍然希望设置一个兜底模型，也可以导出：

```bash
export OPENROUTER_MODEL='openai/gpt-5-mini'
```

### 5.5 前端 API 地址

前端默认代码中：

```ts
process.env.NEXT_PUBLIC_API_BASE_URL || "/api"
```

所以分两种情况：

#### 情况 A：本机开发，推荐

不设置 `NEXT_PUBLIC_API_BASE_URL`，让前端走 `/api` 代理。

这种方式最稳，尤其适合本机开发。

#### 情况 B：局域网访问

如果你要让其他电脑、手机通过局域网访问，推荐设置为：

```bash
export NEXT_PUBLIC_API_BASE_URL='http://你的本机IP:8000'
```

例如：

```bash
export NEXT_PUBLIC_API_BASE_URL='http://192.168.0.101:8000'
```

## 11. 模型切换说明

当前项目里模型有两个层级：

### 11.1 启动环境变量层

例如：

```bash
export OPENROUTER_MODEL='openai/gpt-5-mini'
```

这是默认值或兜底值。

### 11.2 平台配置层

在平台“Prompt 配置中心 / 全局模型配置”中，可以选择当前生效模型，例如：

- `openai/gpt-5-mini`
- `openai/gpt-5.3-chat`
- `claude-sonnet-4.6`

当前逻辑下：

- **新提交任务**
- **新重试任务**

优先使用平台当前启用模型。

所以如果你启动参数里写的是：

```bash
export OPENROUTER_MODEL='openai/gpt-5-mini'
```

但平台当前启用的是：

```text
openai/gpt-5.3-chat
```

那么新任务实际执行的还是 `openai/gpt-5.3-chat`。

## 12. 局域网访问常见问题

### 12.1 页面显示 `Failed to fetch`

优先检查：

1. 前端是否设置了正确的 API 地址

```bash
export NEXT_PUBLIC_API_BASE_URL='http://你的本机IP:8000'
```

2. 后端是否监听 `0.0.0.0`

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

3. 局域网内其他设备能否直接访问：

```text
http://你的本机IP:8000/health
```

### 12.2 本机可成功，局域网提交任务失败

优先检查：

1. 是否有多个 worker 同时监听同一个 Redis 队列
2. 是否有错误目录下的 worker 在跑
3. API 和 worker 是否都从当前项目的 `backend/` 目录启动

建议执行：

```bash
cd /Users/yanshaohui/Documents/GitHub/xhsocr/backend
source .venv/bin/activate
python worker.py
```

并确保只保留这一套 worker。

## 13. OpenRouter 测试方式

如果怀疑大模型调用异常，可以直接在 `backend` 目录测试：

```bash
cd /Users/yanshaohui/Documents/GitHub/xhsocr/backend
source .venv/bin/activate
export OPENROUTER_API_KEY='你的可用 key'
```

测试 `gpt-5-mini`：

```bash
curl -i https://openrouter.ai/api/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $OPENROUTER_API_KEY" \
  -d '{"model":"openai/gpt-5-mini","messages":[{"role":"user","content":"ping"}]}'
```

测试 `claude-sonnet-4.6`：

```bash
curl -i https://openrouter.ai/api/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $OPENROUTER_API_KEY" \
  -d '{"model":"claude-sonnet-4.6","messages":[{"role":"user","content":"ping"}],"reasoning":{"enabled":true}}'
```

如果返回 `HTTP 200`，说明 key 和模型可用。
